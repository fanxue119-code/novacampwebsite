# Nova 官网 · 技术需求附录

> 本文档给**技术团队**看，是对原 PRD（`nova-camp-prd.pdf`）的增量补充与会议调整记录。
> 开发排期、阶段划分见配套文档《Nova 官网项目推进手册》。
> 版本：v1.2（SKU / 拼团规则待饭团最终确认，相关字段暂不实现）
> 上线目标：2026 年 5 月中下旬

---

## ⚠️ 重要说明

本文档中标注 🟡 **"待饭团确认"** 的部分，说明业务规则尚未最终敲定。技术侧**不要提前按推测规则实现**，以避免后续返工。建议的处理方式：

- **可以做**：搭好数据结构的壳（字段先留，可空）、留好扩展点
- **先别做**：写死具体的业务逻辑（如次卡组合包、拼团档位数量、分销返现规则）
- **等规则到了再做**：对应 Phase 3、Phase 4 再按最终规则实现

具体死线：
- 次卡 / 全科全套规则：饭团 W2 末交付
- 拼团 / 分销规则：饭团 W4 末交付

## 三、已确认的业务规则

### 3.1 SKU 体系（大方向已定，细节待确认）

**整期课**（🟡 待定）
- 含ROBBIE版 / 不含ROBBIE版两档
- 具体场次与价格：🟡 待饭团最终给

**次卡**（🟡 规则待饭团 W2 末交付）
- 已知方向：1 次 / 4 次 / 5 次 / 10 次
- 已知约束：
  - 只能选普通老师（不含 700 元档）
  - 不参与拼团
  - 可分销，但无返现
- 🟡 **待饭团确认**：具体价格、组合包规则、全科全套促销规则、700 元档老师名单
- 📌 技术建议：SKU 表结构先按通用模型搭，**具体档位与组合 Phase 3 再落地**

**700 元档老师**（✅ 已定）
- 单独售卖，不进次卡

**价格策略**（✅ 已定）
- 中英双语页面，价格写死不做汇率换算

**场地配额**（✅ 已定）
- 无限制
- 单堂课上限 800 人

### 3.2 拼团（核心模型已定，细节待确认）

**✅ 已确认的核心差异**（与原 PRD 不同，以此为准）
- **取消"团长"概念**，官网不公开 10/20 人团档位
- 先到先得，售完关闭对应档位
- **分销商持固定链接**，链接绑定"团"，自动归团
- 填团顺序：先填 1 号团，满员进 2 号，依次
- 无"开团/等待/超时"逻辑
- 仅整期课可参与拼团，次卡不可

**🟡 待饭团 W4 末确认**
- 档位数量（初步方向：200 × 20 人团 + 200 × 10 人团，**数字待最终敲定**）
- 每档价格 / 差价
- 分销返现规则（金额 / 比例 / 结算时间 / 自动 or 人工）
- 分销商名单与链接分配方式

📌 **技术建议**：Phase 4 前先不要按初步方向写死档位数量和分销逻辑。Phase 4 启动前会拿到最终规则。

### 3.3 支付（✅ 已定）

**国内：支付宝手机网站支付**
- 跳转模式
- 以后端异步通知为准

**海外：Wise**
- 展示 Wise 收款账户 + 用户上传打款凭证
- 后台人工确认到账后手动改订单状态为 PAID
- 后台需有"待确认"订单列表 + 确认按钮

### 3.4 核销（✅ 已定）

**整期课**
- 扫码核销一次

**次卡**
- 购买时选老师和课程
- 每次核销对应一次扣减
- 核销关联"老师 + 课次"
- 防重复核销

### 3.5 结算（✅ 已定）
- 饭团与峰子直接结算，不过公司账
- 系统只负责数据记录和导出，不做账务流

## 四、订单状态机（✅ 已定）

```
INIT
 ├─► PAYING ─┬─► PAID ─┬─► CHECKED_IN
 │           │         └─► REFUNDED
 │           └─► PAYMENT_FAILED
 │
 └─► GROUP_PENDING ─┬─► GROUP_SUCCESS ─► PAID ─► CHECKED_IN
                    └─► GROUP_FAILED ─► REFUNDED

CLOSED（超时/手动关闭）
```

## 五、数据表建议

### 5.1 可以先搭的基础表（原 PRD 已有）

- `users`
- `camp_sessions`
- `camp_skus`（基础字段先搭，见下）
- `orders`
- `order_items`
- `payments`
- `teachers`
- `timetables`
- `checkins`

### 5.2 字段建议（按确认程度分类）

**✅ 现在就可以定字段的**

`camp_skus`
- `sku_type`：整期 / 次卡（枚举先定好两个值）
- `joinable_group`：是否可拼团（boolean，次卡固定 false）

`checkins`
- `teacher_id`
- `class_date`

`order_items`
- `remaining_times`：次卡剩余次数（整期课为 NULL）

**🟡 结构先留，具体值等饭团规则**

`camp_skus`
- `card_times`：次卡次数（1/4/5/10）——字段可以加，具体档位等 Phase 3
- `teacher_id`：次卡绑定老师——字段可以加，绑定逻辑 Phase 3 再写
- 组合包如何建模：🟡 等饭团 W2 末的组合规则再定，可能需要额外的 `sku_bundles` 表

**🟡 暂不建议建表 / 先空**

`groups`、`group_members`
- 原 PRD 有这两张表，但本期拼团模型与原 PRD 差异很大（团池 + 自动归团）
- 🟡 **建议 Phase 4 启动前（W4 末拿到饭团规则后）再设计这两张表的字段**
- 不要现在就按"pool_index 1–200"写死

`referrals`（分销归属）
- 🟡 建议 Phase 4 启动前再设计
- 可以先在 `orders` 表预留 `referral_code` 字段（varchar，允许空），后续填充

## 六、非功能需求（✅ 已定）

| 项 | 要求 |
|---|---|
| 报名流程 | ≤ 5 步 |
| H5 底部 | 固定 CTA |
| 支付状态 | 以后端异步通知为准 |
| 核销码 | 随机 token / 签名串，不暴露顺序 ID |
| 后台权限 | 按角色鉴权 |
| 关键操作 | 留日志（改状态、退款、手动确认支付等）|
| 核销端 | 支持弱网 / 短时离线队列 + 重试 |
| 并发 | 单场 800 人并发核销需通过压测 |
