# Nova 官网技术需求 Sprint 拆解

> 基于 `nova-prd-v2.md` 按 Phase 划分 6 个 Sprint，输出 Markdown 表格。

---

## 总览：各 Sprint 一句话重点

| Sprint | 日期范围 | 核心目标 | 最关键交付物 |
|---|---|---|---|
| Sprint 0 | 4/20 - 4/26 | 技术选型 + 环境就绪 | 技术栈定下来 + classes 表结构 |
| Sprint 1 | 4/27 - 5/3 | 官网能看能填表 | 首页/老师/课表/表单 + bundles 课包定义 |
| Sprint 2 | 5/4 - 5/10 | 能下单能支付能核销 | 订单状态机 + 支付宝/Wise + 核销端 |
| Sprint 3 | 5/11 - 5/24 | 次卡能卖能核销 | promotions 次卡规则绑定课包 |（与 Sprint 4 并行）
| Sprint 4 | 5/11 - 5/24 | 拼团分销能跑通 | 5人/10人/20人团档位 + 分销链接 |
| Sprint 5 | 5/25 - 5/30 | 上线 | 压测 + 培训 + 发布 |

**昨天会议对 Sprint 的核心影响**：把原来"平铺 SKU"改成"课→包→促销"三层模型，次卡和拼团不再是独立 SKU，而是同一课包的不同促销方式。
Sprint 3 和 Sprint 4 都是 promotions 表的不同 promotion_type，后端可以共用同一套表结构和接口框架，前端页面互相独立，所以并行开发可行，拼团档位和分销名单能否在 5/11 前给出初版

## 底层模型说明（课→包→促销）

```
classes（课，最小单元）
  ↓ 组合成
bundles（课包：整期A=23节课 / 整期B=19节课 / 次卡=1节课...）
  ↓ 绑定促销产生
skus（可售单元：课包 + 促销方式 + 价格）
  ↓ 用户购买生成
orders → order_items → checkins
```

**促销类型（promotion_type）**：
- `single`：单买价（不设库存）
- `group_5`：5人公开团（不设库存）
- `group_10`：10人团（仅整期A，不对外显示，按团数设库存）
- `group_20`：20人团（仅整期A，不对外显示，按团数设库存）
- `class_pass`：次卡（1/4/5/10次，不设库存）

**库存规则**：
- `group_5`：后台可配置最大团数（如 200个团 = 1000人），默认很大
- `group_10` 和 `group_20`：按团数设库存（如 20个10人团 + 10个20人团）
- `single` / `class_pass` / 单天：均不设库存

**核销规则**：
- 整期A/B：每天扫一次码，核销当天6节课，发当日手环
- 单天：扫一次码，核销当天6节课，发当日手环
- 次卡：每次上课扫一次码，核销选中的单节课
- 防重复：同一天/同一课次不可重复核销

**class_id 关联规则**：
- 整期A/B、单天：购买时自动关联 bundle 内所有固定 class_id，按天核销对应6节
- 次卡：购买后用户在 classes 列表选单节，核销时关联具体 class_id
