## 三、已确认的业务规则

### 3.1 产品模型：课 → 包 → 促销（已确认）

底层结构统一为三层：

```
classes（课，最小单元：1节课）
  ↓ 组合成
bundles（课包，内部概念，不对客户展示）
  ↓ 绑定促销产生
skus（可售单元：课包 + 促销方式 + 价格）
```

**课包类型（bundles）**：
- **整期A**：23天固定课表（每天6节课），含/不含 Robbie 版两档
- **整期B**：19天固定课表（23天-4节课-1个老师），含/不含 Robbie 版两档
- **单天**：1天6节课固定课表
- **次卡基础包**：可选池（用户在写死的课表上选择）

**促销类型（promotions，绑定课包产生 SKU）**：
- `single`：单买价
- `group_5`：5人公开团（库存后台可配置，默认很大）
- `group_10`：10人团（仅整期A，不对外显示，按团数设库存）
- `group_20`：20人团（仅整期A，不对外显示，按团数设库存）
- `class_pass`：次卡（1/4/5/10次）

**库存规则**：
- 仅 `group_10` 和 `group_20` 按团数设固定库存
- `group_5` 库存后台可配置（如 200个团），默认很大
- `single` / `class_pass` / 单天：均不设库存

**价格策略**（✅ 已定）
- 中英双语页面，价格写死不做汇率换算

**场地配额**（✅ 已定）
- 场地可容纳 1000 人
- 单堂课上限 800 人

### 3.2 拼团（核心模型已定，细节待确认）

拼团 = 课包 + 拼团促销。5人/10人/20人团是同一课包的不同促销档位。

**✅ 已确认的核心差异**（与原 PRD 不同，以此为准）
- **取消"团长"概念**，官网仅公开 5 人团档位
- 10人团/20人团不对外显示，仅通过分销商固定链接进入
- **分销商持固定链接**，链接绑定"团"，自动归团
- 填团顺序：先填 1 号团，满员进 2 号，依次
- 无"开团/等待/超时"逻辑
- 仅整期课可参与拼团，次卡不可

**库存规则**：
- 5人团：库存后台可配置（如 200个团），默认很大，满了自动开下一团
- 10人团/20人团：按团数设固定库存（如 20个10人团 + 10个20人团）

**🟡 待饭团 W4 末确认**
- 10人团/20人团档位数量最终确定
- 每档价格 / 差价
- 分销返现规则（金额 / 比例 / 结算时间 / 自动 or 人工）
- 分销商名单与链接分配方式

📌 **技术建议**：Phase 4 前先不要按初步方向写死档位数量和分销逻辑。Phase 4 启动前会拿到最终规则。

### 3.3 支付（✅ 已定）

**国内：支付宝手机网站支付**
- 跳转模式
- 以后端异步通知为准

**海外：Wise**
- 展示 Wise 收款账户 + 用户上传打款凭证
- 后台人工确认到账后手动改订单状态为 PAID
- 后台需有"待确认"订单列表 + 确认按钮

### 3.4 核销（✅ 已定）

**核销方式**
- **整期A/B**：每天扫一次码，核销当天 6 节课，发当日手环（当天内有效，无需重复扫码）
- **单天**：扫一次码，核销当天 6 节课，发当日手环
- **次卡**：每次上课扫一次码，核销选中的单节课

**防重复核销**
- 同一天/同一课次不可重复核销
- 已核销当天显示"已发手环，当日有效"

### 3.5 结算（✅ 已定）
- 饭团与峰子直接结算，不过公司账
- 系统只负责数据记录和导出，不做账务流

### 3.6 用户购买流程（✅ 已定）

```
[用户端 - 官网 H5]                              [后端]
   │                                               │
   ▼                                               ▼
首页 Landing ──GET /api/bundles──────────────────► 返回课包列表
   │                                               │
   ▼                                               ▼
课程详情页 ──GET /api/skus?bundle_id=xxx─────────► 返回该课包的所有促销档位
   │                                               │
   ▼                                               ▼
选择 SKU（课包+促销）────────────────────────────► 校验库存（拼团购需校验）
   │                                               │
   ▼                                               ▼
报名表单 ──POST /api/orders──────────────────────► 创建订单（状态 INIT）
   │                                               │
   ▼                                               ▼
支付方式选择 ──POST /api/payments────────────────► 支付宝跳转 / Wise 展示账户
   │                                               │
   ▼                                               ▼
支付成功回调 ◄──异步通知────────────────────────── 改订单状态为 PAID
   │                                               │
   ▼                                               ▼
我的订单 / 核销码 ◄─GET /api/orders/{id}────────── 返回订单+核销二维码
   │                                               │
   ▼                                               ▼
[现场 - 核销端]
   │
   ▼
安保扫码 ──POST /api/checkins────────────────────► 核销当天课程，发手环
   │                                               │
   ▼                                               ▼
已核销 ◄────────────────────────────────────────── 返回"已发手环，当日有效"
```

**流程说明**：
1. 用户浏览课包（bundles），选择课包后看到该课包的所有促销档位（skus）
2. 选择 SKU 后填写报名表单，提交创建订单
3. 选择支付方式完成付款，订单状态变为 PAID
4. 在"我的订单"中查看核销二维码
5. 现场安保人员扫码核销，整期/单天按天核销发手环，次卡按次核销

## 四、订单状态机（✅ 已定）

```
INIT
 ├─► PAYING ─┬─► PAID ─┬─► CHECKED_IN
 │           │         └─► REFUNDED
 │           └─► PAYMENT_FAILED
 │
 └─► GROUP_PENDING ─┬─► GROUP_SUCCESS ─► PAID ─► CHECKED_IN
                    └─► GROUP_FAILED ─► REFUNDED

CLOSED（超时/手动关闭）
```

## 五、数据表建议（按新模型：classes → bundles → skus → promotions）

### 5.1 基础表（Sprint 0-1 可搭建）

- `users`
- `teachers`
- `timetables`
- `camp_sessions`
- `classes`（课次单元，最小粒度）
- `bundles`（课包定义，内部概念）
- `bundle_classes`（课包与课的关联表）
- `skus`（可售单元：课包 + 促销组合）
- `orders`
- `order_items`
- `payments`
- `checkins`

### 5.2 字段建议（按确认程度分类）

**✅ 现在就可以定字段的**

`classes`
- `date`：课程日期
- `teacher_id`：授课老师
- `session_name`：课程主题/时段

`bundles`
- `bundle_type`：full_a / full_b / single_day / class_pass
- `robbie_included`：是否含 Robbie 版（boolean）
- `total_days`：总天数（23/19/1）

`skus`
- `bundle_id`：关联课包
- `promotion_id`：关联促销规则（可为空，先留）
- `price`：售价
- `joinable_group`：是否可拼团（boolean）

`checkins`
- `order_item_id`：关联订单项
- `class_date`：核销日期
- `class_ids`：当天核销的 class_id 列表（JSON 或关联表）
- `status`：已核销 / 未核销

**🟡 结构先留，具体值等饭团规则**

`promotions`
- `promotion_type`：single / group_5 / group_10 / group_20 / class_pass
- `card_times`：次卡次数（1/4/5/10），class_pass 时必填
- `group_size`：团人数（5/10/20），拼团时必填
- `inventory_type`：inventory_none / inventory_by_group
- `inventory_count`：库存数量（团数或人数）
- `bundle_id`：绑定课包

`order_items`
- `remaining_times`：次卡剩余次数（整期课为 NULL）
- `selected_class_ids`：次卡已选课程（JSON，次卡必填）

**🟡 暂不建议建表 / 先空**

`groups`、`group_members`
- 原 PRD 有这两张表，但本期拼团模型与原 PRD 差异很大（团池 + 自动归团）
- 🟡 **建议 Phase 4 启动前（W4 末拿到饭团规则后）再设计这两张表的字段**
- 不要现在就按"pool_index 1–200"写死

`referrals`（分销归属）
- 🟡 建议 Phase 4 启动前再设计
- 可以先在 `orders` 表预留 `referral_code` 字段（varchar，允许空），后续填充
